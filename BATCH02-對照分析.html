BATCH02.cbl 程式分析與 Java 轉換規劃報告

分析日期: 2026-06-03
分析對象: BATCH02.cbl (20 年老程式)
目標: 評估改寫成 Java 的可行性與風險

1. 業務邏輯解析

1.1 程式核心功能

這是一支每日批次對帳/統計程式，在每晚 01:30 執行，針對特定交易類型進行累計與費用計算。

1.2 變數含義與業務意義

輸入記錄結構 (80 bytes)

欄位 位置 長度 COBOL 型別 業務含義 範例值
WK-FLD1 1-8 8 PIC X(8) 交易序號/帳號 10000001
WK-AMT1 9-21 13 PIC 9(11)V99 交易金額 0000350000000 = $35,000.00
WK-CD1 22-25 4 PIC X(4) 交易類型代碼 A01 (含空格)
WK-FLD4 26-33 8 PIC X(8) 交易日期 20260603

關鍵業務代碼

  • 'A01 ': 特定交易類型（可能是特許交易/主要業務線/需扣費交易）
  • 只有這類交易需要進入子程序 3000-SUBR
  • 需要計算手續費/稅費（WK-AMT1 * K001）

常數 K001 = 0.021

  • 費率 2.1%（可能是手續費率、稅率、或利潤分成比例）
  • 2022-07-01 調整過（歷史記錄顯示 K001 ADJUSTED）
  • 可能原本是 2.0% 或其他數值

累計器

變數 業務含義
CNT-A 所有交易筆數
WK-SUM1 所有交易總金額
CNT-B A01 交易筆數（特許交易計數）
WK-SUM2 A01 交易總金額
WK-SUM3 A01 交易的費用累計（金額 × 2.1%）

1.3 程式流程圖

0000-MAIN
  │
  ├─→ 1000-INIT (開啟檔案)
  │
  ├─→ 2000-PROC (循環處理每筆交易)
  │     │
  │     ├─→ 讀取記錄到 WK-REC
  │     ├─→ CNT-A += 1
  │     ├─→ WK-SUM1 += WK-AMT1
  │     │
  │     └─→ IF WK-CD1 = 'A01 '
  │           └─→ 3000-SUBR
  │                 ├─→ CNT-B += 1
  │                 ├─→ WK-SUM2 += WK-AMT1
  │                 └─→ WK-SUM3 += (WK-AMT1 × 0.021)
  │
  └─→ 9000-WRAP (關閉檔案 + 顯示統計)
        └─→ DISPLAY 五項統計結果

2.1 規則定義

規則編號: BR-BATCH02-001
規則名稱: 交易處理與分類統計
版本: 2022-07-01 (K001 調整版)

輸入規格:
  - 檔案格式: 固定長度 80 bytes
  - 欄位定義:
      序號: 1-8 (文字)
      金額: 9-21 (數值, 格式 9(11)V99, 單位: 分/cent)
      類型代碼: 22-25 (文字, 含尾隨空格)
      日期: 26-33 (YYYYMMDD)

  R1-全量統計:
    條件: 每筆記錄
    動作:
      - CNT-A += 1
      - WK-SUM1 += 交易金額

  R2-特許交易處理:
    條件: 類型代碼 == 'A01 ' (含尾隨空格)
    動作:
      - CNT-B += 1
      - WK-SUM2 += 交易金額
      - WK-SUM3 += (交易金額 × 0.021)

  R3-費率常數:
    名稱: K001
    數值: 0.021 (2.1%)
    適用: 僅 A01 類型交易
    歷史: 2022-07-01 調整

  R4-類型代碼定義:
    A01: 特許交易 (需計費)
    A02: 一般交易類型 2
    A03: 一般交易類型 3
    A04: 一般交易類型 4
    A05: 一般交易類型 5
    註: 非 A01 類型不進入費用計算

輸出規格:
  - 顯示五項統計:
      CNT-A: 總交易筆數
      WK-SUM1: 總交易金額
      CNT-B: A01 筆數
      WK-SUM2: A01 總金額
      WK-SUM3: A01 費用總額 (金額 × 2.1%)

錯誤處理:
  - 檔案無法開啟: 顯示錯誤碼並終止
  - 檔案讀取失敗: 顯示錯誤碼並終止

3. COBOL → Java 轉換路徑

3.1 必須完整保留的行為

A. 精度要求

⚠️ 關鍵風險點

  • 金額計算必須使用 BigDecimal（避免浮點數誤差）
  • 小數位數保持 2 位（ROUND_HALF_UP）
  • K001 常數必須精確到 0.021（不能用 float/double）
// ❌ 錯誤做法
double k001 = 0.021;  // 會有浮點誤差
double sum = amount * k001;

// ✅ 正確做法
BigDecimal K001 = new BigDecimal("0.021");
BigDecimal sum = amount.multiply(K001).setScale(2, RoundingMode.HALF_UP);

B. 字串比對邏輯

⚠️ 陷阱: WK-CD1 = 'A01 ' 含尾隨空格

// ❌ 錯誤做法
if (typeCode.trim().equals("A01"))  // 會遺漏正確交易

// ✅ 正確做法
if ("A01 ".equals(typeCode))  // 完整比對含空格

C. 檔案處理

  • 固定長度 80 bytes 記錄切割
  • 需處理 ASCII/EBCDIC 編碼差異（若原系統是 mainframe）
  • 保留檔案狀態檢查邏輯（file status '00', '10'）

D. 錯誤處理

  • 檔案開啟失敗需立即終止（不繼續執行）
  • 輸出格式需一致（下游可能 parse 這些 DISPLAY 輸出）

3.2 影響下游系統的風險點

風險 1: 輸出格式變更 🔴 高風險

現況:
- 原程式用 DISPLAY 輸出到 SYSOUT
- 下游可能有 JCL 抓取這些統計值（用 IEBGENER 或 REXX 解析）

風險:
- 改變輸出格式 → 監控系統/報表程式斷鏈
- 數字前導零改變 → 解析失敗

建議:
- 保持完全相同的輸出格式
- 或在轉換前調查所有使用 BATCH02 輸出的系統

風險 2: 檔案格式假設 🟡 中風險

需確認:
- INPUT02.DAT 來自哪個上游系統？
- 上游是否持續產生固定 80-byte 格式？
- 是否有 EBCDIC → ASCII 編碼轉換需求？

建議:
- 與上游系統 owner 確認格式契約
- 考慮支援多種格式（但不破壞現有邏輯）

風險 3: 排程整合 🟡 中風險

現況:
- 原程式在 JCL PROD.JCLLIB(BATCH02J) 中執行
- 每晚 01:30 自動執行

轉 Java 後需處理:
- JCL 需修改為呼叫 Java（或改用 cron/Spring Batch scheduler）
- JCL 中可能有:
- COND 檢查（return code 判斷）
- Dataset allocation
- 依賴其他 JCL step

建議:
- 先取得完整 JCL BATCH02J 內容分析
- 考慮保留 return code 規範（0=成功, 4/8/12=各種錯誤）

風險 4: K001 常數管理 🟢 低風險但需決策

現況:
- 寫死在程式碼（VALUE 0.021）
- 2022 年改過一次需要重新編譯部署

選項:
| 方案 | 優點 | 缺點 |
|------|------|------|
| 保持寫死 | 簡單、與原邏輯一致 | 改動需重新編譯部署 |
| application.properties | 無需重啟可改 | 需建立配置管理流程 |
| 資料庫 table | 可審計、可版控 | 增加複雜度、DB 依賴 |

建議:
- 先保持寫死（降低轉換風險）
- 第二階段優化時改為外部配置

2.1 Record Parser

public class FixedLengthRecordParser {
    public TransactionRecord parse(String line) {
        if (line.length() != 80) {
            throw new IllegalArgumentException("Record must be 80 bytes");
        }

        String accountNo = line.substring(0, 8);
        String amountStr = line.substring(8, 21);  // 13 chars
        String typeCode = line.substring(21, 25);  // 4 chars (含空格!)
        String transDate = line.substring(25, 33);

        // COBOL PIC 9(11)V99 → BigDecimal
        BigDecimal amount = new BigDecimal(amountStr)
            .movePointLeft(2);  // 移動小數點 2 位

        return new TransactionRecord(accountNo, amount, typeCode, transDate);
    }
}

2.2 Transaction Processor (核心邏輯)

@Service
public class TransactionProcessor {
    private static final BigDecimal K001 = new BigDecimal("0.021");
    private static final String A01_TYPE = "A01 ";  // 含空格!

    public BatchSummary process(List<TransactionRecord> records) {
        BatchSummary summary = new BatchSummary();

        for (TransactionRecord rec : records) {
            // R1: 全量統計 (對應 2000-PROC)
            summary.incrementCntA();
            summary.addToSum1(rec.getAmount());

            // R2: A01 特許交易處理 (對應 3000-SUBR)
            if (A01_TYPE.equals(rec.getTypeCode())) {
                summary.incrementCntB();
                summary.addToSum2(rec.getAmount());

                // 費用計算: WK-SUM3 = WK-SUM3 + (WK-AMT1 * K001)
                BigDecimal fee = rec.getAmount()
                    .multiply(K001)
                    .setScale(2, RoundingMode.HALF_UP);
                summary.addToSum3(fee);
            }
        }

        return summary;
    }
}

2.3 主程式入口

@SpringBootApplication
public class Batch02Application implements CommandLineRunner {

    @Autowired
    private TransactionFileReader fileReader;

    @Autowired
    private TransactionProcessor processor;

    public static void main(String[] args) {
        SpringApplication.exit(SpringApplication.run(Batch02Application.class, args));
    }

    @Override
    public void run(String... args) throws Exception {
        try {
            // 1000-INIT
            List<TransactionRecord> records = fileReader.readAll("INPUT02.DAT");

            // 2000-PROC
            BatchSummary summary = processor.process(records);

            // 9000-WRAP
            printSummary(summary);

            System.exit(0);  // Return code 0

        } catch (Exception e) {
            System.err.println("ERR: " + e.getMessage());
            System.exit(8);  // Return code 8 (error)
        }
    }

    private void printSummary(BatchSummary s) {
        System.out.println("----- SUMMARY -----");
        System.out.printf("CNT-A   : %07d%n", s.getCntA());
        System.out.printf("WK-SUM1 : %014.2f%n", s.getSum1());
        System.out.printf("CNT-B   : %07d%n", s.getCntB());
        System.out.printf("WK-SUM2 : %014.2f%n", s.getSum2());
        System.out.printf("WK-SUM3 : %014.2f%n", s.getSum3());
    }
}

一、業務功能解析

核心業務

華南銀行(HNCB)房貸帳戶日結批次 - 呆帳損失準備計提系統

這支程式每日凌晨 01:30 執行,處理當日所有帳戶快照,主要目的是:
1. 統計全行帳戶本金餘額
2. 針對房貸帳戶計提呆帳損失準備金(這是核心業務)
3. 產出會計分錄所需數據

變數實際含義

輸入資料格式 (80 字元固定長度)

位置 1-8   : WK-FLD1  = 客戶代號 (例: 10000001)
位置 9-21  : WK-AMT1  = 本金餘額 (例: 0000350000000 = 3,500,000.00 元)
位置 22-25 : WK-CD1   = 帳戶類別 (例: 'A01 ' 含尾部空白)
位置 26-33 : WK-FLD4  = 帳務日期 (例: 20260603)

關鍵變數

  • WK-AMT1: 帳戶本金餘額(元),PIC 9(11)V99,最大 999億元
  • WK-CD1: 帳戶類別代號
  • 'A01 ' = 房貸帳戶 ← 唯一觸發呆帳提存的類別
  • 'A02 ' = 信用貸款
  • 'A03 ' = 車輛貸款
  • 'A04 ' = 信用卡
  • 'A05 ' = 活期儲蓄存款
  • K001 = 0.021: 呆帳提存準備率 = 2.1%
  • 這是依金管會/IFRS 9 預期信用損失(ECL)評估設定的提存比率
  • 不是房貸利率(利率重算由 RATE-ADJ 系統處理)
  • 2022-07-01 最後調整

累計變數

  • WK-SUM1: 全帳戶本金總餘額(所有類別)
  • WK-SUM2: 房貸帳戶本金總餘額(僅 A01)
  • WK-SUM3: 房貸呆帳提存準備金總額 ← 核心輸出
  • CNT-A: 總帳戶數
  • CNT-B: 房貸帳戶數

為什麼是 0.021 (2.1%)?

這是呆帳損失準備提存率,不是利率:
- 依據 IFRS 9 預期信用損失(ECL)模型
- 金管會要求銀行對放款資產提列損失準備
- 房貸屬於擔保放款,風險相對較低,提存率 2.1%
- 這筆準備金會計分錄為:
借: 提存費用(損益表費用科目) 貸: 備抵呆帳(資產負債表備抵科目)

A. 必須完整保留的行為

1. 數值精度 (CRITICAL)

// COBOL: PIC 9(11)V99 → Java: BigDecimal
BigDecimal principalBalance = new BigDecimal("3500000.00");
BigDecimal provisionRate = new BigDecimal("0.021");
BigDecimal provision = principalBalance.multiply(provisionRate)
    .setScale(2, RoundingMode.HALF_UP);  // 必須四捨五入到小數 2 位

風險: Java double 會有浮點誤差,必須用 BigDecimal

2. 字串比對邏輯 (CRITICAL)

// COBOL: IF WK-CD1 = 'A01 ' (含尾部空白)
String accountType = record.substring(21, 25);  // 必須取 4 字元
if ("A01 ".equals(accountType)) {  // 必須含尾部空白
    calculateProvision();
}

風險: Java 字串 trim() 會移除空白,導致比對失敗

3. 累計順序與初始值

// 必須初始化為 0
BigDecimal sumAll = BigDecimal.ZERO;
BigDecimal sumMortgage = BigDecimal.ZERO;
BigDecimal sumProvision = BigDecimal.ZERO;
int countAll = 0;
int countMortgage = 0;

// 必須按順序累計
for (AccountRecord record : records) {
    countAll++;
    sumAll = sumAll.add(record.getPrincipalBalance());

    if ("A01 ".equals(record.getAccountType())) {

        BigDecimal provision = record.getPrincipalBalance()
            .multiply(PROVISION_RATE)
            .setScale(2, RoundingMode.HALF_UP);

    }
}

public class BigDecimalUtil {
    private static final int SCALE = 2;
    private static final RoundingMode ROUNDING_MODE = RoundingMode.HALF_UP;

    public static BigDecimal calculateProvision(BigDecimal principal, BigDecimal rate) {
        return principal.multiply(rate).setScale(SCALE, ROUNDING_MODE);

    public static BigDecimal sum(BigDecimal... values) {
        return Arrays.stream(values)
            .reduce(BigDecimal.ZERO, BigDecimal::add);

總結

這支 20 年的 COBOL 程式是華南銀行房貸呆帳提存的核心系統,每日產出會計分錄所需的準備金金額。轉換到 Java 時,數值精度和字串比對邏輯是最大風險點,必須使用 BigDecimal 並保留尾部空白。建議採用 Spring Batch 框架,外部化 K001 常數,並與下游 5 個系統進行完整整合測試。

關鍵風險提醒

  1. 財務影響: WK-SUM3 的任何誤差都會直接影響財報損益
  2. 法規遵循: K001 提存率必須符合金管會/IFRS 9 規範
  3. 系統相依: 5 個下游系統依賴此程式輸出,改動需全面測試
  4. 精度要求: 必須使用 BigDecimal,double 會導致分毫差異累積

建議轉換順序

  1. Phase 1: 建立 Java 平行系統,雙軌運行驗證
  2. Phase 2: 逐日比對 COBOL vs Java 輸出,確保 100% 一致
  3. Phase 3: 下游系統整合測試 (至少 3 個月)
  4. Phase 4: 正式切換,保留 COBOL 作為備援 (6 個月)
